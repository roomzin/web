Technical Deep Dive

How Roomzin achieves sub-millisecond searches at scale—without magic

The Engineering Behind the Speed

Roomzin isn't faster because of complex optimizations—it's faster because of deliberate simplicity. Here's what makes it different.

Core Architectural Innovations

1

Fixed Schema = Bitmap Magic

By limiting amenities to 32 and cancellation rules to 8, we encode them as u32 and u8 bitmasks.

The Impact:

  • • Amenity filtering = bitwise AND operation (single CPU cycle)
  • • No string comparisons, no hash lookups
  • • SIMD-able for multiple properties at once
2

String Elimination

All strings are converted to numeric IDs at ingest time:

"pool" → bit position 2
"Hilton" → 16-bit ID: 142
"deluxe_suite" → 8-bit code: 7

Result: 50% memory reduction vs. raw JSON

3

H3 Hexagonal Grid Search

Uses Uber's H3 library for O(1) radius searches instead of O(log n) R-tree lookups. 

// Convert lat/long to H3 cell (resolution 8)
let cell = latlng.to_cell(Resolution::Eight);

// Get k-ring neighbors (k=3 = ~3km radius)
let neighbors = cell.grid_disk::>(3);

// Search = bitwise AND of pre-computed cell bitmaps
// Result: 0.6ms P95 for ~3km radius search
4

Segment-Based Parallelism

Data is partitioned into segments (e.g., "paris_north", "paris_south") that operate independently.

Each segment has its own processing queue

Paris' 1,800 hotels → 5 segments = 5x CPU utilization

No cross-segment locking or coordination

5

Write Priority Streams

QUIC protocol with priority channels ensures writes never starve, even at 25k RPS searches.

Write Stream Priority Channel
Read Streams Best-Effort

Result: Booking commits in <1ms even during search storms

6

Lock-Free Snapshots

Snapshots use Rust's atomic references, copy-on-write, and cold-snapshot rebuild for zero-downtime persistence.

Snapshot Pause Time 4-10ms
60M Records Load Time 2-8 seconds
Snapshot Size vs Raw Data ~15%

Deliberate Trade-offs

Roomzin's performance comes from accepting specific constraints:

Constraint

Max 32 amenities, 8 cancellation rules

Benefit

Bitmask encoding enables SIMD and single-cycle filtering

Constraint

Hot window only (e.g., 60 days)

Benefit

Fits entirely in RAM, predictable memory usage

Constraint

Custom binary protocol (not HTTP/JSON)

Benefit

Zero parsing overhead, 10x smaller payloads

Constraint

Full replication in cluster (no sharding)

Benefit

No cross-node joins, consistent 1-2ms reads on followers

These constraints cover 95%+ of booking platform use cases. For edge cases, Roomzin gracefully falls back to the database.

Decoupled Request/Response Channels

Unlike traditional request/response blocking, Roomzin uses fully decoupled channels for maximum throughput.

How It Works

Client-side: Each request gets a unique correlation ID. SDK maintains request/response mapping.

Server-side: Separate threads handle requests and responses. No blocking, pure async processing.

Result: 10k RPS with predictable latency, even with mixed request sizes

Enterprise Context: Shard Isolation

In large booking platforms, data is sharded by region or zone. Each shard runs its own Roomzin cluster.

Shard = Independent Cluster

Each geographical region runs its own Roomzin cluster with full HA. SDKs only need to know their shard's nodes.

No Cross-Shard Communication

Like Redis deployments, each shard is isolated. Traffic stays within the shard for lowest latency.
